# Аудит репозитория (черновик, 2026-05-26)

Краткая фиксация состояния и рисков на момент приведения монорепо к модульным PR.

## Положительное

- Чёткое разделение `apps/` и `packages/`, Supabase-миграции вынесены в `supabase/`.
- Продуктовые ограничения по приватности live и координатам friends отражены в правилах репозитория (`.cursorrules` / `project-guide.md`).
- Примеры `.env.example` для web и mobile с разделением публичных и серверных ключей.

## Риски / пробелы

- Автотесты (unit / E2E) не охватывают основной поток карты и RLS — стоит планировать.
- Полный UI авторизации и часть синхронизации онбординга с БД могут быть в процессе — сверять с последним `main`.
- Lockfile политики: в корневом `.gitignore` игнорируется `pnpm-lock.yaml` — это осознанный выбор команды или стоит пересмотреть для воспроизводимости CI.

## Рекомендации

1. Включить в CI хотя бы `pnpm typecheck` и `pnpm lint`.
2. Периодически прогонять `supabase db lint` / проверку политик RLS при изменениях схемы.
3. Документировать в `docs/` фактический процесс релиза mobile (EAS профили).

*Документ носит информационный характер и не заменяет формальный security review.*
